## 一、模块概述

系统设置是题小助 V2.0 的基础支撑模块，涵盖平台全局配置、RBAC 权限体系、操作审计、字典数据、通知渠道及数据备份等核心功能。该模块为其他所有业务模块提供统一的配置读取、权限校验和日志记录能力，是平台安全性和可运维性的基石。

核心功能包括：

- **系统配置管理**：基于 key-value 模型的全局配置，按模块分组，支持 Redis 缓存加速
- **RBAC 权限体系**：角色 -> 菜单 -> 权限三级层次，动态菜单加载，前端 `v-permission` 指令控制按钮级权限
- **菜单管理**：树形结构菜单，支持目录/菜单/按钮三种类型，前端路由动态注入
- **权限管理**：权限编码规范 `module:sub:action`，支持 CRUD 及角色批量分配
- **数据权限**：行级数据过滤，支持全部/本部门及以下/本部门/仅本人四种范围
- **操作日志**：中间件自动记录，审计日志不可修改和删除，保留 180 天
- **字典管理**：全局 key-value 字典数据，Redis 缓存，前端通过 `DictSelect`/`DictTag` 组件消费
- **通知设置**：支持站内信/邮件/短信/微信四通道，模板引擎支持变量替换
- **数据备份**：mysqldump 全量 + binlog 增量，30 天保留策略，定时任务自动执行

本模块复用现有的分层架构（Controller -> Service -> Repository -> Model），通过包级函数调用，依赖全局 `database.DB` 和 `database.RDB` 进行数据访问和缓存。

**请求处理流程**：

```
客户端请求
    │
    ▼
Router (路由匹配)
    │
    ▼
Middleware 链
    ├── CORS (跨域)
    ├── RateLimit (限流)
    ├── JWT Auth (从 Header 提取 token，验证签名，注入 user_id)
    ├── RBAC Permission (检查角色/权限)
    ├── DataPermission (行级数据权限过滤)
    └── OperationLog (异步记录操作日志到数据库)
    │
    ▼
Controller
    ├── 参数绑定 (ShouldBindJSON / ShouldBindQuery)
    ├── 参数校验 (binding tag)
    └── 调用 service 包级函数
    │
    ▼
Service
    ├── 业务逻辑校验
    ├── 调用 repository 包级函数
    ├── Redis 缓存读写 (菜单/权限/字典)
    ├── Model <-> DTO 转换
    └── 日志记录
    │
    ▼
Repository
    ├── GORM 查询构建
    ├── 分页查询 (Count + Offset + Limit)
    └── CRUD 操作
    │
    ▼
MySQL (system_settings, roles, menus, permissions, operation_logs, ...)
```

## 五、数据库表

系统设置模块涉及跨多个数据库设计文档的表，涵盖权限管理、系统配置、日志审计、通知和备份等功能域。

### 5.1 权限管理表

```
roles (角色表)
  │
  ├── N:M ──→ menus (菜单表, 通过 role_menus 关联)
  │              │
  │              └── type: 1=目录, 2=菜单, 3=按钮
  │              └── permission 字段存储权限标识
  │
  ├── N:M ──→ permissions (权限表, 通过 role_permissions 关联)
  │              │
  │              └── code: module:sub:action 格式
  │              └── type: 1=菜单, 2=按钮, 3=API
  │
  ├── N:M ──→ users (用户表, 通过 user_roles 关联)
  │
  └── button_permissions (按钮权限表, 关联 menus)
```

```
data_permissions (数据权限表)
  └── type: 1=全部, 2=本部门及以下, 3=本部门, 4=仅本人

depts (部门表, 树形结构)
  └── 用于数据权限的部门过滤
```

### 5.7 关键索引设计

| 表名 | 索引 | 类型 | 用途 |
|------|------|------|------|

## 七、核心实现

### 7.1 系统配置管理

#### 7.1.1 Key-Value 配置模型

系统配置采用通用 key-value 模型，存储在 `system_settings` 表中。所有业务配置（站点信息、班级设置、资源设置、功能开关等）统一通过该表管理，避免为每类配置单独建表。

```go
// model/system.go
type SystemSetting struct {
    ID        uint      `gorm:"primarykey" json:"id"`
    Key       string    `gorm:"uniqueIndex;size:100;not null" json:"key"`
    Value     string    `gorm:"type:text" json:"value"`
    Remark    string    `gorm:"size:200" json:"remark"`
    CreatedAt time.Time `json:"created_at"`
    UpdatedAt time.Time `json:"updated_at"`
}
```

#### 7.1.2 按模块分组查询

Service 层按业务模块定义 key 列表，实现分组查询：

```go
// service/system/system_service.go - GetClassSettings

func GetClassSettings() (*dto.ClassSettings, error) {
    var settings []model.SystemSetting
    keys := []string{
        "class_max_students", "class_allow_self_join", "class_require_approval",
        "class_default_type", "class_enable_code", "class_code_length",
    }
    database.DB.Where("`key` IN ?", keys).Find(&settings)

    // 构建 key-value 映射，解析为结构化 DTO
    kv := make(map[string]string)
    for _, s := range settings {
        kv[s.Key] = s.Value
    }
    // ... 类型转换填充 DTO
}
```

#### 7.1.3 Upsert 更新策略

配置更新使用 `FirstOrCreate` + `Assign` 模式，key 存在则更新 value，不存在则创建：

```go
// service/system/system_service.go - UpdateSettings

func UpdateSettings(settings map[string]string) error {
    for key, value := range settings {
        setting := model.SystemSetting{Key: key, Value: value}
        if err := database.DB.Where("key = ?", key).
            Assign(setting).FirstOrCreate(&setting).Error; err != nil {
            return fmt.Errorf("更新设置 %s 失败: %w", key, err)
        }
    }
    return nil
}
```

#### 7.1.4 Redis 缓存层

高频读取的配置项（如站点名称、功能开关）应缓存到 Redis，更新时主动失效：

```
缓存 Key 格式: setting:{key}
过期时间: 1 小时 (3600s)
更新策略: Write-Through (写入 DB 后立即删除缓存)
```

读取流程：
```
Service.GetSetting(key)
    │
    ├── Redis GET setting:{key}
    │     ├── 命中 → 返回缓存值
    │     └── 未命中 ↓
    │
    ├── MySQL SELECT FROM system_settings WHERE key = ?
    │
    ├── Redis SET setting:{key} = value (EX 3600)
    │
    └── 返回值
```

### 7.2 RBAC 权限体系

#### 7.2.1 角色-菜单-权限层次

RBAC 权限体系采用三级层次结构：

```
Role (角色)
  │
  ├── N:M ──→ Menu (菜单)
  │              ├── type=1: 目录 (路由分组)
  │              ├── type=2: 菜单 (页面路由)
  │              └── type=3: 按钮 (权限标识)
  │
  └── N:M ──→ Permission (权限)
                 └── code: module:sub:action
                     例: question:public:view
                         exam:paper:create
                         user:list:delete
```

用户通过 `user_roles` 关联表获得角色，角色通过 `role_menus` 和 `role_permissions` 关联表获得菜单和权限。

#### 7.2.2 动态菜单加载流程

前端登录后动态加载菜单和权限的完整流程：

```
1. 用户登录 → 获取 JWT Token

2. 前端调用 GET /api/v1/menus/routes
   │
   ▼
3. MenuController.GetUserRoutes(userID)
   │
   ▼
4. menu_service.GetUserMenus(userID)
   │
   ├── 查询用户所有角色
   │   userRepo.FindByID(userID) → user.Roles
   │
   ├── 遍历角色，查询每个角色的菜单
   │   userRepo.FindMenusByRoleID(role.ID)
   │
   ├── 合并去重（排除按钮类型 type≠3）
   │
   └── 构建菜单树 buildMenuTree()
       │
       ▼
5. transformToRoutes() 转换为前端路由格式
   返回 [{name, path, component, redirect, meta, children}]

6. 前端 permissionStore 接收路由
   │
   ├── router.addRoute() 动态注入路由
   │
   └── 缓存到内存
```

#### 7.2.3 权限中间件实现

权限中间件从 Redis 缓存读取用户权限集，逐条匹配：

```go
// middleware/permission.go

func PermissionMiddleware(requiredPermission string) gin.HandlerFunc {
    return func(c *gin.Context) {
        userID := c.MustGet("user_id").(uint)

        // 从 Redis 缓存获取用户权限
        permissions, err := getUserPermissions(userID)

        // 逐条匹配
        for _, p := range permissions {
            if p == requiredPermission {
                c.Next()
                return
            }
        }

        response.Error(c, http.StatusForbidden, "无权限访问")
        c.Abort()
    }
}
```

权限缓存策略：

```
缓存 Key:  user:permissions:{userID}
数据结构:  Redis SET (权限编码集合)
过期时间:  5 分钟 (300s)
失效时机:  角色变更、菜单变更时主动清除
```

#### 7.3.1 树形结构

```go
// model/user.go 中的 Menu 模型
type Menu struct {
    ID        uint      `gorm:"primarykey" json:"id"`
```

#### 7.3.3 前端 v-permission 指令

按钮级权限通过自定义指令 `v-permission` 控制：

```typescript
// src/directives/permission.ts
import { useUserStore } from '@/stores/user'

export const permission = {
  mounted(el: HTMLElement, binding: { value: string | string[] }) {
    const userStore = useUserStore()
    const requiredPermissions = Array.isArray(binding.value)
      ? binding.value
      : [binding.value]

    const hasPermission = requiredPermissions.some(p =>
      userStore.permissions.includes(p)
    )

    if (!hasPermission) {
      el.parentNode?.removeChild(el)
    }
  }
}

// 使用示例
// <el-button v-permission="'question:public:create'">新增</el-button>
// <el-button v-permission="['exam:paper:create', 'exam:paper:edit']">编辑</el-button>
```

```go
func DataPermissionMiddleware(tableName string) gin.HandlerFunc {
    return func(c *gin.Context) {
        userID := c.MustGet("user_id").(uint)
        roleIDs := c.MustGet("role_ids").([]uint)

        // 管理员拥有全部权限
        if isAdmin(roleIDs) {
            c.Next()
            return
        }

        // 查询用户的数据权限配置
        var dp model.DataPermission
        database.DB.Where("user_id = ?", userID).First(&dp)

        switch DataPermissionType(dp.Type) {
        case DataPermissionAll:
            // 不添加过滤条件
        case DataPermissionDept, DataPermissionDeptSub:
            // 查询部门及下级部门 ID 列表
            deptIDs := getDeptIDs(dp.DeptID, dp.Type == DataPermissionDeptSub)
            condition := fmt.Sprintf("%s.dept_id IN ?", tableName)
            c.Set("data_permission_condition", condition)
            c.Set("data_permission_args", []interface{}{deptIDs})
        case DataPermissionSelf:
            condition := fmt.Sprintf("%s.created_by = ?", tableName)
            c.Set("data_permission_condition", condition)
            c.Set("data_permission_args", []interface{}{userID})
        }

        c.Next()
    }
}
```

Repository 层应用数据权限：

### 7.6 操作日志

#### 7.6.1 中间件自动记录

操作日志通过 Gin 中间件自动记录，无需业务代码手动调用：

```go
// middleware/operation_log.go

func OperationLogMiddleware() gin.HandlerFunc {
    return func(c *gin.Context) {
        startTime := time.Now()

        // 读取并恢复请求体（后续处理器需要读取）
        var body []byte
        if c.Request.Body != nil {
            body, _ = io.ReadAll(c.Request.Body)
            c.Request.Body = io.NopCloser(bytes.NewBuffer(body))
        }

        c.Next()

        // 请求处理完成后记录日志
        latency := time.Since(startTime).Milliseconds()

        // 异步写入数据库（通过 MQ 或 goroutine）
        go func() {
            opLog := &model.OperationLog{
                UserID:      getUserID(c),
                Username:    getUsername(c),
                Module:      parseModule(c.Request.URL.Path),
                Action:      parseAction(c.Request.Method),
                Method:      c.Request.Method,
                Path:        c.Request.URL.Path,
                IP:          c.ClientIP(),
                UserAgent:   c.Request.UserAgent(),
                Status:      parseStatus(c.Writer.Status()),
                Latency:     latency,
            }
            logService.CreateOperationLog(opLog)
        }()
    }
}
```

| 字段 | 来源 | 说明 |
|------|------|------|
| `user_id` | JWT Token 解析 | 操作人 ID |

#### 7.6.3 审计日志不可变性

审计日志的安全要求：

1. **写入后不可修改**：`operation_logs` 表不提供 UPDATE 接口
2. **写入后不可删除**：管理端不提供删除单条日志的 API
3. **保留 180 天**：定时任务 `log_cleanup.go` 中清理策略应调整为 180 天
4. **覆盖敏感操作**：管理员查看他人私有资源、重置密码、分配角色等操作必须记录

```go
    key := fmt.Sprintf("notify_limit:%s:%s", channel, scene)
    count, _ := database.RDB.Incr(context.Background(), key).Result()

        database.RDB.Expire(context.Background(), key,
            time.Duration(limit.Period)*time.Second)
    }

    return count <= int64(limit.MaxCount)
}
```

```go
// service/system/system_service.go - CreateBackup

func CreateBackup(req *dto.CreateBackupRequest) (*model.BackupRecord, error) {
    // 1. 创建备份记录 (status=pending)
    record := &model.BackupRecord{
        Type:   req.Type,
        Status: "pending",
    }
    database.DB.Create(record)

    // 2. 异步执行备份（通过 MQ 或 goroutine）
    go func() {
        // 更新状态为 running
        database.DB.Model(record).Update("status", "running")
        database.DB.Model(record).Update("started_at", time.Now())

        // 执行 mysqldump
        filePath := generateBackupFilePath(config)
        err := executeMysqldump(filePath, config)

        if err != nil {
            database.DB.Model(record).Updates(map[string]interface{}{
                "status":        "failed",
                "error_message": err.Error(),
            })
            return
        }

        // 记录文件大小
        fileSize := getFileSize(filePath)
        database.DB.Model(record).Updates(map[string]interface{}{
            "status":       "completed",
            "file_path":    filePath,
            "file_size":    fileSize,
            "completed_at": time.Now(),
        })

        // 更新配置的最后备份时间
        database.DB.Model(&model.BackupConfig{}).
            Where("id = ?", req.ConfigID).
            Update("last_backup_at", time.Now())
    }()

    return record, nil
}
```

### 8.1 缓存 Key 总览

| 缓存用途 | Key 格式 | 数据结构 | 过期时间 | 说明 |
|----------|----------|----------|----------|------|
| 用户菜单 | `menu:user:{userID}` | STRING (JSON) | 24 小时 | 用户菜单树缓存 |
| 角色菜单 | `menu:role:{roleID}` | STRING (JSON) | 24 小时 | 角色菜单列表缓存 |
| 菜单树 | `menu:tree` | STRING (JSON) | 24 小时 | 完整菜单树缓存 |
| 用户权限 | `user:permissions:{userID}` | SET (权限编码) | 5 分钟 | 用户权限码集合 |
| 系统配置 | `setting:{key}` | STRING | 1 小时 | 系统配置值缓存 |

| Token 黑名单 | `token:blacklist:{jti}` | STRING | Token 剩余有效期 | 已注销的 JWT Token |

### 8.2 缓存失效策略

```
系统配置更新:
  MySQL UPDATE → Redis DEL setting:{key}

角色/菜单变更:
  MySQL UPDATE → Redis DEL menu:role:{roleID}
              → 遍历该角色所有用户 → Redis DEL menu:user:{userID}
              → Redis DEL user:permissions:{userID}

字典数据更新:
  MySQL UPDATE → Redis DEL dict:{type_code}

功能开关更新:
  MySQL UPDATE → Redis DEL feature:{key}
```

### 8.3 缓存穿透防护

对于不存在的 Key，缓存空值防止穿透：

```go
// 查询配置时，若数据库不存在则缓存空字符串
func GetSetting(key string) (string, error) {
    cached, err := cache.Get("setting:" + key)
    if err == nil {
        if cached == "" {
            return "", errors.New("配置不存在")
        }
        return cached, nil
    }

    var setting model.SystemSetting
    err = database.DB.Where("key = ?", key).First(&setting).Error
    if err != nil {
        cache.Set("setting:"+key, "", 300) // 缓存空值 5 分钟
        return "", err
    }

    cache.Set("setting:"+key, setting.Value, 3600)
    return setting.Value, nil
}
```

## 十、安全考虑

### 10.1 审计日志安全

| 措施 | 实现方式 |
|------|----------|
| 写入后不可修改 | `operation_logs` 表不提供 UPDATE API，Service 层不暴露修改方法 |
| 写入后不可删除（管理端） | 管理端仅提供 `clean` 接口清理过期日志，不提供删除单条 API |
| 保留 180 天 | 定时任务 `LogCleanupTask` 设置 180 天清理阈值 |
| 敏感操作全覆盖 | 中间件记录所有请求，特殊操作（重置密码、权限变更）在 Service 层补充详细描述 |
| 管理员资源访问审计 | 管理员查看他人私有资源时，Service 层额外写入操作日志 |

### 10.2 权限分配约束

| 约束 | 实现方式 |
|------|----------|
| 不能分配自己没有的权限 | 角色分配时校验分配者的权限集是否包含被分配角色的所有权限 |
| 超级管理员不受约束 | 超级管理员（role_id=1）跳过权限校验 |
| 系统角色不可删除 | `role.IsSystem = true` 时拒绝删除操作 |
| 有用户关联的角色不可删除 | 删除前检查 `user_roles` 表中该角色的用户数 |

### 10.3 安全响应头

通过中间件统一设置安全响应头：

```go
// 中间件中添加
c.Header("X-Content-Type-Options", "nosniff")
c.Header("X-Frame-Options", "DENY")
c.Header("Strict-Transport-Security", "max-age=31536000")
c.Header("Content-Security-Policy", "default-src 'self'")
```

### 10.4 敏感配置保护

| 配置类型 | 保护措施 |
|----------|----------|
| 数据库密码 | 配置文件 `config.yaml` 中存储，不通过 API 返回完整值 |
| 邮件密码 | `email_configs.password` 字段 API 返回时脱敏（`******`） |
| 短信密钥 | `sms_configs.access_key_secret` 字段 API 返回时脱敏 |
| 存储密钥 | `storage_configs.secret_key` 字段 API 返回时脱敏 |
| 备份加密 | 备份文件使用 AES-256 加密，密钥单独管理 |

### 10.5 操作限流

| 接口 | 限流策略 |

## 十一、错误处理

### 11.1 错误码定义

复用 `pkg/errors/` 中的 `AppError` 体系，系统设置模块扩展以下业务错误码：

| 错误码 | HTTP 状态 | 场景 |
|--------|-----------|------|
| `A0001` | 400 | 参数错误（缺少必填字段、格式不正确） |
| `A0002` | 401 | 未认证（JWT 缺失或过期） |
| `A0003` | 403 | 无权限（非管理员访问管理端接口） |
| `A0004` | 404 | 资源不存在（菜单、角色、配置项） |
| `A3001` | 400 | 业务错误：系统内置角色不可删除 |
| `A3002` | 400 | 业务错误：角色下存在用户，无法删除 |
| `A3003` | 400 | 业务错误：存在子菜单，无法删除 |
| `A3004` | 400 | 业务错误：权限编码已存在 |
| `A3005` | 400 | 业务错误：功能开关不存在 |
| `A3006` | 400 | 业务错误：备份记录不存在 |
| `A3007` | 400 | 业务错误：只能恢复已完成的备份 |
| `A3008` | 400 | 业务错误：不能删除正在执行的备份 |
| `A3009` | 400 | 业务错误：配置键已存在 |
| `A3010` | 400 | 业务错误：字典类型编码已存在 |
| `A3011` | 400 | 业务错误：字典数据值在该类型下已存在 |

### 11.2 统一响应格式

所有接口通过 `pkg/response/` 返回统一 JSON 信封：

```json
// 成功响应
{"code": "00000", "msg": "success", "data": {...}}

// 分页响应
{"code": "00000", "msg": "success", "data": {"list": [...], "total": 100, "page": 1, "page_size": 10}}

// 错误响应
{"code": "A3001", "msg": "系统内置角色不可删除", "data": null}
```

### 11.3 Service 层错误处理模式

```go
// Controller 层
func (ctrl *SystemController) UpdateFeatureFlag(c *gin.Context) {
    key := c.Param("key")
    var req dto.UpdateFeatureRequest
    if err := c.ShouldBindJSON(&req); err != nil {
        response.Error(c, http.StatusBadRequest, "参数错误: "+err.Error())
        return
    }

    if err := system.UpdateFeatureFlag(key, &req); err != nil {
        response.Error(c, http.StatusInternalServerError, err.Error())
        return
    }
    response.SuccessWithMessage(c, "更新成功", nil)
}

// Service 层
func UpdateFeatureFlag(key string, req *dto.UpdateFeatureRequest) error {
    var flag model.FeatureFlag
    if err := database.DB.Where("`key` = ?", key).First(&flag).Error; err != nil {
        if errors.Is(err, gorm.ErrRecordNotFound) {
            return fmt.Errorf("功能开关不存在: %s", key)
        }
        return fmt.Errorf("查询功能开关失败: %w", err)
    }
    // ...
}
```
