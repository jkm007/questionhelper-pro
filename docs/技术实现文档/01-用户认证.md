## 一、模块概述

用户认证模块是题小助平台的安全基石，涵盖以下核心功能：

- **用户注册**：支持用户名注册，邮箱必填（需验证码校验），手机号选填，图形验证码防机器人，注册成功后自动登录
- **用户登录**：支持账号密码登录、邮箱验证码登录、微信小程序/公众号/扫码登录、GitHub/Google OAuth 登录；含登录限制（密码错误锁定、IP 封禁、图形验证码升级）
- **Token 管理**：JWT access + refresh token 模式，Redis 黑名单机制，自动静默刷新，角色切换重新签发
- **密码找回**：邮箱验证码验证身份后重置密码，重置后所有设备强制退出
- **第三方登录（OAuth）**：支持微信（小程序/公众号/扫码）、GitHub、Google，首次登录绑定或创建账号
- **图形验证码**：基于 base64Captcha 生成数字/字母/算术验证码，Redis 存储，单次有效
- **设备管理**：登录设备记录、当前设备标识、踢出指定/所有设备
- **安全日志**：记录登录、退出、密码变更、绑定/解绑等安全事件
- **账号注销**：条件检查、冷静期 30 天、可撤销、数据清理策略

**请求处理流程**：

```
客户端请求
    │
    ▼
Router (路由匹配)
    │
    ▼
Middleware 链
    ├── CORS 中间件 (跨域处理)
    ├── RateLimit 中间件 (IP 限流)
    ├── Auth 中间件 (JWT 解析 + 黑名单检查)
    ├── Permission 中间件 (RBAC 权限校验)
    └── OperationLog 中间件 (操作日志记录)
    │
    ▼
Controller (参数绑定 + 校验)
    │
    ▼
Service (业务逻辑 + 事务管理)
    │
    ▼
Repository (GORM 数据库操作)
    │
    ▼
Model (数据库实体)
```

## 五、数据库表

用户认证模块涉及以下数据表：

| 表名 | 说明 | 关键字段 | 关联关系 |
|------|------|----------|----------|
| `users` | 用户主表 | id, username (unique), password (bcrypt), email, phone, status, login_fail_count, lock_until, register_source, last_login_at, last_login_ip, logout_at | 主表，被所有认证表引用 |
| `user_privacy` | 用户隐私设置 | user_id (unique FK→users), profile_visible, realname_visible, email_visible, stats_visible, class_visible | 1:1 关联 users |
| `oauth_users` | 第三方登录绑定 | user_id (FK→users), provider, provider_type, provider_user_id, access_token, refresh_token | N:1 关联 users，一个用户可绑定多个第三方 |
| `login_devices` | 登录设备记录 | user_id (FK→users), device_id, device_type, device_name, browser, os, ip, location, token_jti, is_current | N:1 关联 users |
| `security_logs` | 安全日志 | user_id (FK→users), event_type (login/logout/password_change/bind/unbind/account_deactivate), ip, user_agent, status | N:1 关联 users |
| `password_history` | 密码历史 | user_id (FK→users), password (bcrypt) | N:1 关联 users，保留最近 3 次 |
| `user_real_names` | 实名认证 | user_id (unique FK→users), real_name, id_card (AES 加密), id_card_hash, status | 1:1 关联 users |
| `role_applications` | 角色申请 | user_id (FK→users), role_id (FK→roles), reason, status | N:1 关联 users 和 roles |
| `roles` | 角色表 | id, name, code (unique), is_system, status | 被 user_roles 多对多关联 |
| `user_roles` | 用户角色关联 | user_id (FK→users), role_id (FK→roles) | 多对多中间表 |

**索引设计**：

- `users` 表：username (unique), email, phone, deleted_at
- `oauth_users` 表：user_id, (provider + provider_user_id) 联合唯一
- `login_devices` 表：user_id, device_id
- `security_logs` 表：user_id, event_type, created_at
- `password_history` 表：user_id

## 七、核心实现

### 7.1 注册流程

**请求入口**：`POST /api/v1/auth/register`

**完整流程**：

```
1. Controller 层 (auth_controller.go)
   ├── 绑定 RegisterRequest DTO + 参数校验
   │   - username: required, 2-50 位
   │   - password: required, 6-20 位
   │   - confirmPassword: 必须与 password 一致
   │   - email: required, email 格式
   │   - agreement: required, 必须为 true
   ├── 调用 Service 层 Register()

2. Service 层 (auth_service.go)
   ├── 2.1 图形验证码校验
   │   ├── 从 Redis 获取 qh:captcha:{captchaID} 对应的验证码
   │   ├── 比较用户输入（忽略大小写）
   │   └── 验证失败返回错误，验证成功删除 Redis Key（单次有效）
   │
   ├── 2.2 邮箱验证码校验
   │   ├── 从 Redis 获取 qh:email:{email} 对应的验证码
   │   └── 验证通过后删除 Key
   │
   ├── 2.3 唯一性校验
   │   ├── 查询 username 是否已存在 → ErrUserExists (A0202)
   │   ├── 查询 email 是否已存在 → ErrUserExists (A0202)
   │   └── 若 phone 非空，查询 phone 是否已存在 → ErrUserExists (A0202)
   │
   ├── 2.4 敏感词检查
   │   └── 检查 nickname 是否包含敏感词（DFA 算法）
   │
   ├── 2.5 创建用户
   │   ├── 密码 bcrypt 加密（cost=12，可配置）
   │   ├── 构造 User model（status=1, register_source=请求来源）
   │   ├── Repository 层 INSERT 到 users 表
   │   └── 同时创建 user_privacy 默认记录
   │
   ├── 2.6 分配默认角色
   │   └── 查询 is_default=true 的 Role，关联到 user_roles
   │
   ├── 2.7 生成 Token 对
   │   ├── Access Token: HS256, payload={user_id, username, role_ids, jti}, TTL=2h
   │   ├── Refresh Token: HS256, payload={user_id, jti, type=refresh}, TTL=7d
   │   └── JTI 使用 crypto/rand 生成 16 字节随机数（hex 编码）
   │
   ├── 2.8 记录登录设备
   │   └── INSERT login_devices（device_id, device_type, ip, user_agent, token_jti）
   │
   ├── 2.9 记录安全日志
   │   └── INSERT security_logs（event_type=register, ip, user_agent, status=1）
   │
   └── 2.10 返回 LoginResponse（access_token, refresh_token, user info）

3. Controller 层
   ├── response.Success(c, loginResponse)
   └── HTTP 200
```

**关键代码路径**：

- DTO 定义：`internal/dto/auth_dto.go` → `RegisterRequest`
- 参数校验：Gin binding tags (`binding:"required,min=2,max=50"`)
- 密码加密：`pkg/encrypt/bcrypt.go` → `HashPassword()`
- 验证码校验：`pkg/captcha/captcha.go` → `VerifyCaptchaFromRedis()`
- 用户创建：`internal/repository/user/user_repo.go` → `CreateUser()`

### 7.2 登录流程

**请求入口**：`POST /api/v1/auth/login`

**完整流程**：

```
1. Controller 层
   ├── 绑定 LoginRequest DTO
   │   - username: required（支持用户名/邮箱/手机号）
   │   - password: required
   │   - captchaId + captchaCode: 可选（密码错误 3 次后必填）
   │   - rememberMe: 可选（延长 Refresh Token 至 30 天）
   └── 调用 Service 层 Login()

2. Service 层 (auth_service.go)
   ├── 2.1 图形验证码校验（若 captchaId 非空）
   │   └── 验证失败返回 A0203 错误
   │
   ├── 2.2 IP 限流检查
   │   ├── Redis Key: qh:login:limit:{ip}
   │   └── 同一 IP 短时间内大量失败 → 临时封禁
   │
   ├── 2.3 用户查找
   │   ├── 优先按 username 查询
   │   ├── 其次按 email 查询
   │   ├── 最后按 phone 查询
   │   └── 未找到 → 返回"用户名或密码错误"（统一提示，防止枚举）
   │
   ├── 2.4 账号状态检查
   │   ├── status=0 (禁用) → ErrAccountDisabled (A0204)
   │   ├── status=2 (注销中) → 返回注销中提示
   │   └── lock_until > now → 返回"账号已锁定，剩余 X 分钟"
   │
   ├── 2.5 密码校验
   │   ├── bcrypt.CompareHashAndPassword(password_hash, input_password)
   │   ├── 失败：
   │   │   ├── login_fail_count++
   │   │   ├── login_fail_count >= 5 → 设置 lock_until = now + 15min
   │   │   ├── login_fail_count >= 3 → 下次登录需要图形验证码
   │   │   └── 更新 users 表 login_fail_count 和 lock_until
   │   └── 成功：login_fail_count = 0, lock_until = NULL
   │
   ├── 2.6 异常登录检测
   │   ├── 检查 IP 地理位置是否变化（对比 last_login_ip）
   │   ├── 检查设备是否变化（对比 last_login_device）
   │   └── 触发安全验证（邮箱验证码）或发送安全通知
   │
   ├── 2.7 生成 Token 对
   │   ├── Access Token: TTL = 7200s (2h)
   │   ├── Refresh Token: TTL = rememberMe ? 2592000s (30d) : 604800s (7d)
   │   └── 生成唯一 JTI
   │
   ├── 2.8 更新用户登录信息
   │   ├── last_login_at = now
   │   ├── last_login_ip = clientIP
   │   └── last_login_device = userAgent
   │
   ├── 2.9 记录登录设备
   │   └── INSERT login_devices（is_current=true，其他设备设为 false）
   │
   ├── 2.10 记录安全日志
   │   └── INSERT security_logs（event_type=login, status=1）
   │
   └── 2.11 返回 LoginResponse

3. 前端处理
   ├── 存储 access_token 到内存/localStorage
   ├── 存储 refresh_token 到 httpOnly Cookie（Secure, SameSite=Strict）
   ├── 获取用户信息 + 角色列表
   └── 根据角色加载动态路由，跳转首页
```

**登录方式扩展**：

- **邮箱验证码登录**：省略密码校验步骤，改用 `qh:email:{email}` 验证码校验
- **微信小程序登录**：`wx.login()` 获取 code → 后端调用微信 API 换 openid → 查找/创建用户 → 生成 Token
- **OAuth 登录**：见 7.5 节

### 7.3 Token 管理

**JWT 结构**：

```json
// Access Token Payload
{
  "user_id": 1,
  "username": "admin",
  "role_ids": [1, 2],
  "jti": "a1b2c3d4e5f6...",  // 16 字节随机 hex
  "exp": 1717234567,          // 过期时间
  "iat": 1717227367,          // 签发时间
  "iss": "questionhelper"     // 签发者
}

// Refresh Token Payload
{
  "user_id": 1,
  "jti": "f6e5d4c3b2a1...",
  "type": "refresh",
  "exp": 1717832167,
  "iat": 1717227367
}
```

**Token 参数**：

| 参数 | Access Token | Refresh Token |
|------|-------------|---------------|
| 签名算法 | HS256 | HS256 |
| 有效期 | 7200s (2 小时) | 604800s (7 天) / 2592000s (30 天，记住我) |
| 存储位置 | 内存 / localStorage | httpOnly Cookie |
| 用途 | API 请求认证 | 静默刷新 Access Token |
| 黑名单 | 退出时加入 Redis 黑名单 | 踢出设备时加入黑名单 |

**Token 刷新流程**：

```
前端请求拦截器 (request.ts / auth.ts)
    │
    ├── 1. 每次请求前检查 Access Token 剩余有效期
    │   └── 剩余 < 5 分钟 → 触发刷新
    │
    ├── 2. 刷新锁机制
    │   ├── isRefreshing 标志位
    │   ├── 刷新中，新请求加入 waitingQueue
    │   └── 刷新完成后，队列中的请求使用新 Token 重发
    │
    ├── 3. 调用 POST /api/v1/auth/refresh
    │   ├── 请求体: { refreshToken: "..." }
    │   ├── 后端验证 Refresh Token 签名 + 过期时间
    │   ├── 检查 Refresh Token JTI 是否在黑名单中
    │   ├── 生成新的 Token 对（新 JTI）
    │   └── 返回 { accessToken, refreshToken, expiresIn }
    │
    └── 4. 刷新失败处理
        ├── Refresh Token 过期或无效
        ├── 清除本地 Token
        └── 跳转登录页
```

**Token 黑名单机制**：

```
退出登录时：
    1. 解析 Access Token 获取 JTI
    2. Redis SET qh:token:bl:{jti} = "1"
    3. TTL = Access Token 剩余有效期（避免无限增长）
    4. 同时使对应 Refresh Token 失效

认证中间件检查：
    1. 解析 JWT 获取 JTI
    2. Redis EXISTS qh:token:bl:{jti}
    3. 存在 → 返回 401 "令牌已失效"
    4. 不存在 → 继续处理

定时清理：
    - task/token_cleanup.go 定期清理已过期的黑名单 Key
    - 由于设置了 TTL，Redis 会自动过期，此任务为兜底清理
```

**角色切换 Token 处理**：

```
1. 用户请求切换角色
2. Service 层验证用户是否拥有目标角色
3. 生成新 Token 对（role_ids 更新为目标角色）
4. 旧 Token 的 JTI 加入 Redis 黑名单
5. 返回新 Token 对
6. 前端更新存储的 Token
```

### 7.4 密码找回

**请求入口**：`POST /api/v1/auth/password/reset-request` + `POST /api/v1/auth/password/reset`

**完整流程**：

```
阶段一：申请密码重置 (reset-request)
    │
    ├── 1. Controller 绑定参数
    │   - account: 用户名/邮箱/手机号
    │   - captchaId + captchaCode: 图形验证码
    │
    ├── 2. 图形验证码校验
    │
    ├── 3. 查找用户
    │   ├── 未找到 → 返回统一提示"验证码已发送"（防止账号枚举）
    │   └── 找到 → 继续
    │
    ├── 4. 发送邮箱验证码
    │   ├── 生成 6 位随机数字验证码
    │   ├── Redis SET qh:email:{email} = code, TTL=300s (5min)
    │   ├── 频率限制：qh:email:limit:{email}, TTL=60s
    │   ├── 每日上限：qh:email:daily:{email}, 计数器, TTL=24h
    │   └── 异步发送邮件（通过 mq/notification_job）
    │
    └── 5. 返回统一响应（无论账号是否存在）

阶段二：执行密码重置 (reset)
    │
    ├── 1. Controller 绑定参数
    │   - account: 用户名/邮箱/手机号
    │   - code: 邮箱验证码
    │   - newPassword: 新密码
    │
    ├── 2. 邮箱验证码校验
    │   ├── Redis GET qh:email:{email}
    │   └── 验证成功后删除 Key
    │
    ├── 3. 查找用户
    │
    ├── 4. 密码历史检查
    │   ├── 查询 password_history 表（最近 3 条）
    │   ├── 逐一 bcrypt 比较新密码
    │   └── 与历史密码相同 → 返回"不能与最近 3 次密码相同"
    │
    ├── 5. 更新密码
    │   ├── bcrypt 加密新密码
    │   ├── UPDATE users SET password = new_hash, password_changed_at = now
    │   └── INSERT password_history
    │
    ├── 6. 全设备强制退出
    │   ├── 查询该用户所有 login_devices
    │   ├── 将所有 token_jti 加入 Redis 黑名单
    │   └── DELETE login_devices WHERE user_id = ?
    │
    └── 7. 记录安全日志
        └── INSERT security_logs (event_type=password_change)
```

### 7.7 邮件验证码

**请求入口**：`POST /api/v1/auth/email/send`

**限流策略**：

```
1. 频率限制（同邮箱 60 秒 1 次）
   ├── Redis Key: qh:email:limit:{email}
   ├── TTL: 60s
   └── 存在则拒绝，返回剩余等待时间

2. 每日上限（同邮箱每天 10 次）
   ├── Redis Key: qh:email:daily:{email}
   ├── 数据类型: INCR 计数器
   ├── TTL: 24h（从第一次发送开始）
   └── 计数 >= 10 则拒绝

3. IP 限制（同 IP 每小时 50 次）
   ├── Redis Key: qh:email:ip:{ip}
   ├── 数据类型: INCR 计数器
   ├── TTL: 1h
   └── 计数 >= 50 则拒绝

4. 验证次数限制（同一验证码最多验证 5 次）
   ├── Redis Key: qh:email:attempts:{email}
   ├── 数据类型: INCR 计数器
   ├── TTL: 300s（与验证码同步）
   └── 计数 >= 5 则验证码失效
```

**发送流程**：

```
1. 频率/上限检查（Redis 操作）
2. 生成 6 位随机数字验证码
3. Redis SET qh:email:{email} = code, TTL=300s
4. 通过消息队列异步发送邮件
   ├── mq.Push(notification_job, { to: email, subject: "验证码", body: code })
   └── notification_job 调用邮件服务发送
5. 返回成功响应
```

**挂载位置**：

### 8.2 限流中间件

**文件**：`internal/middleware/rate_limit.go`

**算法**：令牌桶（Token Bucket）

```
配置参数：
    rate  = 10  (每秒补充 10 个令牌)
    burst = 20  (桶容量 20 个令牌)

处理逻辑：
    1. 根据客户端 IP 获取 visitor 对象
    2. 计算上次请求到现在的经过时间
    3. 补充令牌: tokens += elapsed_seconds * rate
    4. 令牌数不超过 burst
    5. tokens > 0 → 允许，tokens--
    6. tokens <= 0 → 拒绝，返回 429

清理机制：
    - 后台 goroutine 每分钟清理 3 分钟未活跃的 visitor
    - 避免内存泄漏
```

**应用范围**：

- 全局限流：所有 `/api/v1/*` 接口
- 登录接口可单独配置更严格的限流（如 5 次/分钟）

### 8.3 CORS 中间件

**文件**：`internal/middleware/cors.go`

**配置**：

```
Access-Control-Allow-Origin: *          (开发环境，生产环境应限制域名)
Access-Control-Allow-Methods: GET, POST, PUT, DELETE, OPTIONS
Access-Control-Allow-Headers: Origin, Content-Type, Authorization
Access-Control-Max-Age: 86400          (预检请求缓存 24 小时)

OPTIONS 请求直接返回 204
```

**生产环境建议**：

- `Access-Control-Allow-Origin` 应限制为具体域名（管理后台域名 + 移动端 H5 域名）
- 可通过配置文件动态设置允许的域名列表

## 九、Redis 缓存设计

### 9.1 认证相关 Key

| Key Pattern | 数据类型 | TTL | 说明 |
|-------------|---------|-----|------|
| `qh:token:bl:{jti}` | String | Access Token 剩余有效期 | Token 黑名单，退出登录时写入 |
| `qh:captcha:{captchaID}` | String | 300s (5min) | 图形验证码答案，验证后删除 |
| `qh:captcha:attempts:{key}` | String | 300s (5min) | 图形验证码错误次数 |
| `qh:email:{email}` | String | 300s (5min) | 邮箱验证码，验证后删除 |
| `qh:email:limit:{email}` | String | 60s | 邮箱验证码发送频率限制 |
| `qh:email:daily:{email}` | String (INCR) | 86400s (24h) | 邮箱验证码每日发送次数 |
| `qh:email:attempts:{email}` | String (INCR) | 300s (5min) | 邮箱验证码验证尝试次数 |
| `qh:email:ip:{ip}` | String (INCR) | 3600s (1h) | 同一 IP 邮箱验证码发送次数 |
| `qh:sms:{phone}` | String | 300s (5min) | 短信验证码 |
| `qh:sms:limit:{phone}` | String | 60s | 短信发送频率限制 |
| `qh:sms:daily:{phone}` | String (INCR) | 86400s (24h) | 短信每日发送次数 |
| `qh:login:limit:{ip}` | String (INCR) | 900s (15min) | IP 登录失败次数限制 |
| `qh:reset:token:{uid}` | String | 900s (15min) | 密码重置令牌 |
| `qh:deactivate:code:{uid}` | String | 300s (5min) | 账号注销确认验证码 |
| `qh:login:fail:{username}` | String (INCR) | 900s (15min) | 账号登录失败次数（与 DB 双写） |

### 9.2 Key 设计原则

- **统一前缀**：所有 Key 使用 `qh:` 前缀，避免与其他应用冲突
- **TTL 自动过期**：所有临时数据都设置 TTL，避免内存泄漏
- **单次有效**：验证码类 Key 验证成功后立即删除
- **计数器模式**：使用 INCR 实现频率限制和次数统计
- **分布式友好**：Redis 存储支持多实例部署

## 十、安全考虑

### 10.1 密码安全

| 措施 | 说明 |
|------|------|
| bcrypt 加密 | cost=12（可配置），每次加密结果不同（含随机盐） |
| 密码强度 | 6-20 位，需包含字母和数字 |
| 历史密码检查 | 新密码不能与最近 3 次历史密码相同 |
| 传输安全 | HTTPS 强制，密码不在 URL 或日志中出现 |

### 10.2 Token 安全

| 措施 | 说明 |
|------|------|
| HS256 签名 | 使用服务端密钥签名，防篡改 |
| 短有效期 | Access Token 2 小时，减少泄露风险 |
| JTI 唯一标识 | 每个 Token 唯一 ID，支持精确黑名单 |
| Redis 黑名单 | 退出即时失效，不依赖过期时间 |
| Refresh Token httpOnly Cookie | 禁止 JS 访问，防 XSS 窃取 |
| Cookie 安全属性 | HttpOnly + Secure + SameSite=Strict |

### 10.3 登录安全

| 措施 | 说明 |
|------|------|
| 密码错误锁定 | 连续 5 次错误锁定 15 分钟 |
| 图形验证码升级 | 密码错误 3 次后需要验证码 |
| IP 限流 | 同一 IP 短时间大量失败触发封禁 |
| 统一错误提示 | "用户名或密码错误"，不区分具体原因 |
| 异常登录检测 | IP/设备变化触发安全验证 |
| 新设备通知 | 新设备首次登录发送安全通知邮件 |

### 10.4 接口安全

| 措施 | 说明 |
|------|------|
| CORS 限制 | 生产环境限制允许的域名 |
| Rate Limiting | 全局令牌桶限流（10/s，burst=20） |
| SQL 注入防护 | GORM 参数化查询 |
| XSS 防护 | 输出转义，Content-Type: application/json |
| CSRF 防护 | SameSite=Strict Cookie |
| 敏感词过滤 | 中间件层拦截，DFA 算法 |

### 10.5 数据安全

| 措施 | 说明 |
|------|------|
| 身份证号加密 | AES 对称加密存储，查询使用 hash 索引 |
| 密码不可逆 | bcrypt 单向哈希，无法还原 |
| 软删除 | 用户数据通过 deleted_at 软删除 |
| 日志脱敏 | 日志中不记录密码、Token 原文 |
| 数据导出 | 注销前可导出个人数据 |

### 10.6 账号注销安全

| 措施 | 说明 |
|------|------|
| 身份验证 | 注销需验证密码或邮箱验证码 |
| 条件检查 | 无进行中考试、无未处理申请等 |
| 30 天冷静期 | 30 天内可撤销，30 天后永久删除 |
| 撤销冷却 | 撤销后 30 天内不可再次申请 |
| 全设备退出 | 注销确认后所有设备强制退出 |

## 十一、错误处理

### 11.1 认证模块错误码

| 场景 | HTTP 状态码 | 错误码 | 消息 |

| Token 在黑名单中 | 401 | - | 令牌已失效 |
| Refresh Token 无效 | 401 | A0231 | 刷新令牌无效或过期 |
| 用户名或密码错误 | 400 | A0203 | 用户名或密码错误 |
| 账号已禁用 | 403 | A0204 | 账号已禁用 |
| 账号已锁定 | 423 | - | 账号已锁定，请 X 分钟后重试 |
| 账号注销中 | 403 | - | 账号正在注销中 |
| 用户名已存在 | 409 | A0202 | 用户名已被注册 |
| 邮箱已存在 | 409 | A0202 | 邮箱已被注册 |
| 手机号已存在 | 409 | A0202 | 手机号已被注册 |
| 图形验证码错误 | 400 | A0203 | 验证码错误 |
| 图形验证码过期 | 400 | - | 验证码已过期 |
| 邮箱验证码错误 | 400 | - | 验证码错误或已过期 |
| 验证码发送频繁 | 429 | - | 请 X 秒后重试 |
| 验证码每日上限 | 429 | - | 今日发送次数已达上限 |
| IP 请求过于频繁 | 429 | - | 请求过于频繁，请稍后再试 |
| 参数错误 | 400 | A0001 | 具体参数校验错误信息 |

| 系统内部错误 | 500 | B0001 | 系统内部错误 |

### 11.2 响应格式

```json
// 成功响应
{
  "code": "00000",
  "msg": "success",
  "data": { ... }
}

// 错误响应
{
  "code": "A0203",
  "msg": "用户名或密码错误",
  "data": null
}

// 分页响应
{
  "code": "00000",
  "msg": "success",
  "data": {
    "list": [],
    "total": 100,
    "page": 1,
    "page_size": 10
  }
}
```

### 11.3 前端错误处理

```
Axios 响应拦截器 (request.ts)
    │
    ├── code === "00000" → 返回 data
    │
    ├── code === "A0200" (未授权) 或 HTTP 401
    │   ├── 尝试用 Refresh Token 刷新
    │   ├── 刷新成功 → 重发原请求
    │   └── 刷新失败 → 清除 Token，跳转登录页
    │
    ├── code === "A0230" (Token 过期)
    │   └── 同上，触发 Token 刷新
    │
    ├── code === "A0231" (Refresh Token 无效)
    │   └── 清除 Token，跳转登录页
    │
    ├── HTTP 429 (请求过于频繁)
    │   └── ElMessage.warning("请求过于频繁，请稍后再试")
    │
    └── 其他错误
        └── ElMessage.error(msg)
```
