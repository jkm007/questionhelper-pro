## 一、模块概述

用户管理模块是题小助平台的核心业务模块，建立在用户认证模块之上，涵盖以下核心功能：

- **用户信息管理**：个人资料查看与修改、头像上传（支持裁剪为正方形）、密码修改（含历史密码校验）、隐私设置（多级可见性控制）
- **实名认证**：身份证号 18 位校验（含 ISO 7064:1983.MOD 11-2 校验位算法）、AES 加密存储、手机验证码 + 身份信息格式自动验证、管理员审核流程
- **角色申请**：创作者/教师角色申请、申请状态机（pending/approved/rejected/revoked）、管理员审批工作流、撤销后 30 天冷却期
- **管理员用户管理**：用户列表查询（多维度筛选）、批量操作（启用/禁用/删除/分配角色）、角色分配（全量替换）、密码重置、用户导出 Excel
- **隐私设置**：个人主页、真实姓名、邮箱、学习统计、班级信息的多级可见性控制（全部可见/仅班级成员/仅自己）

**请求处理流程**：

```
客户端请求
    │
    ▼
Router (路由匹配)
    │
    ▼
Middleware 链
    ├── CORS 中间件 (跨域处理)
    ├── RateLimit 中间件 (IP 限流)
    ├── Auth 中间件 (JWT 解析 + 黑名单检查)
    ├── Permission 中间件 (RBAC 权限校验)
    ├── DataPermission 中间件 (数据权限过滤，管理端)
    ├── OperationLog 中间件 (操作日志记录)
    └── SensitiveFilter 中间件 (敏感词过滤，信息修改时)
    │
    ▼
Controller (参数绑定 + 校验)
    │
    ▼
Service (业务逻辑 + 事务管理)
    │
    ▼
Repository (GORM 数据库操作)
    │
    ▼
Model (数据库实体)
```

## 五、数据库表

用户管理模块涉及以下数据表：

| 表名 | 说明 | 关键字段 | 关联关系 |
|------|------|----------|----------|
| `users` | 用户主表 | id, username (unique), password (bcrypt), nickname, email, phone, avatar, gender, birthday, bio, status, real_name, id_card (AES 加密), is_real, register_source, last_login_at, last_login_ip | 主表，被所有关联表引用 |
| `user_privacy` | 用户隐私设置 | user_id (unique FK→users), profile_visible, realname_visible, email_visible, stats_visible, class_visible | 1:1 关联 users |
| `user_real_names` | 实名认证记录 | user_id (unique FK→users), real_name, id_card (AES 加密), id_card_hash (索引), status (0待审/1已认证/2失败), reject_reason, reviewed_by, reviewed_at | 1:1 关联 users |
| `role_applications` | 角色申请记录 | user_id (FK→users), role_id (FK→roles), reason, status (0待审/1通过/2驳回), review_note, reviewed_by, reviewed_at | N:1 关联 users 和 roles |
| `roles` | 角色表 | id, name, code (unique), description, is_default, is_system, sort, status | 被 user_roles 多对多关联 |
| `user_roles` | 用户角色关联 | user_id (PK, FK→users), role_id (PK, FK→roles) | 多对多中间表 |

**索引设计**：

- `users` 表：username (unique), email, phone, deleted_at, (status + deleted_at) 联合索引
- `user_privacy` 表：user_id (unique)
- `user_real_names` 表：user_id (unique), id_card_hash (用于唯一性校验), status
- `role_applications` 表：user_id, role_id, status, (user_id + role_id + status) 联合索引（用于检查同一角色是否有 pending 申请）
- `user_roles` 表：(user_id + role_id) 联合主键

**字段说明**：

- `users.id_card`：AES 加密存储，使用独立于 JWT 的加密密钥；API 响应中脱敏显示（如 `110***********1234`）
- `user_real_names.id_card`：AES 加密存储；`id_card_hash` 使用 SHA-256 生成，用于唯一性校验（查询时先比对 hash，避免解密比对）
- `user_privacy` 可见性值：1=全部可见, 2=仅班级成员, 3=仅自己
- `role_applications.status`：0=待审核(pending), 1=已通过(approved), 2=已驳回(rejected)
- `users.status`：0=禁用, 1=正常, 2=注销中

#### 7.2.1 提交实名认证

**请求入口**：`POST /api/v1/user/realname`

**完整流程**：

```
1. Controller 层 (realname_controller.go)
   ├── 绑定 RealNameSubmitRequest DTO
   │   - real_name: required, 2-50 字符
   │   - id_card: required, 18 位身份证号

2. Service 层 (realname_service.go)
   ├── 2.1 检查是否已认证
   │   ├── Repository: SELECT * FROM user_real_names WHERE user_id=?
   │   ├── 已认证 (status=1) → 返回"已完成实名认证"
   │   ├── 待审核 (status=0) → 返回"认证审核中，请耐心等待"
   │   └── 已驳回 (status=2) → 允许重新提交
   │
   ├── 2.2 身份证号 18 位格式校验
   │   ├── 长度必须为 18 位
   │   ├── 前 17 位必须为数字
   │   ├── 第 18 位必须为数字或 X
   │   ├── 校验出生日期合法性（YYYYMMDD，不早于 1900 年，不晚于当前日期）
   │   ├── 校验地址码合法性（前 6 位为有效行政区划代码）
   │   └── 校验位算法验证（ISO 7064:1983.MOD 11-2）
   │       ├── 加权因子: [7,9,10,5,8,4,2,1,6,3,7,9,10,5,8,4,2]
   │       ├── 前 17 位数字乘以对应加权因子求和
   │       ├── 和 mod 11 得到余数
   │       ├── 余数映射校验位: {0:'1',1:'0',2:'X',3:'9',4:'8',5:'7',6:'6',7:'5',8:'4',9:'3',10:'2'}
   │       └── 校验位不匹配 → 返回"身份证号格式错误"
   │
   ├── 2.3 身份证号唯一性校验
   │   ├── 计算 id_card_hash = SHA-256(id_card)
   │   ├── Repository: SELECT COUNT(*) FROM user_real_names WHERE id_card_hash=? AND user_id!=?
   │   └── 已存在 → 返回"该身份证号已被其他账号认证"
   │
   ├── 2.4 AES 加密身份证号
   │   ├── 调用 pkg/encrypt/aes.go → Encrypt(id_card)
   │   └── 使用独立于 JWT 的 AES 密钥（配置文件中 aes_key 字段）
   │
   ├── 2.5 创建/更新实名认证记录
   │   ├── 若为驳回后重新提交：
   │   │   └── Repository: UPDATE user_real_names SET real_name=?, id_card=?, id_card_hash=?, status=0, reject_reason='', updated_at=? WHERE user_id=?
   │   └── 若为首次提交：
   │       └── Repository: INSERT INTO user_real_names (user_id, real_name, id_card, id_card_hash, status, submitted_at) VALUES (...)
   │
   ├── 2.6 自动审核（V2.0 简化流程）
   │   ├── 实名认证通过手机号验证码 + 身份信息格式验证，无需人工审核
   │   ├── 更新 status=1 (已认证)
   │   ├── 同步更新 users 表：real_name=?, id_card=?, is_real=true
   │   └── 记录审核信息：reviewed_by=系统, reviewed_at=now
   │
   └── 2.7 返回认证结果

3. Controller 层
   └── response.Success(c, realNameResponse)
```

**关键代码路径**：

- DTO 定义：`internal/dto/realname_dto.go` → `RealNameSubmitRequest`
- 身份证校验：`internal/service/user/realname_service.go` → `validateIdCard(idCard)` 内部函数
- AES 加密：`pkg/encrypt/aes.go` → `Encrypt(plaintext)` / `Decrypt(ciphertext)`
- 哈希计算：`crypto/sha256` 标准库

**身份证校验位算法详细实现**：

```go
// validateIdCardCheckDigit 校验身份证号第18位校验位
// 算法: ISO 7064:1983.MOD 11-2
func validateIdCardCheckDigit(idCard string) bool {
    weights := []int{7, 9, 10, 5, 8, 4, 2, 1, 6, 3, 7, 9, 10, 5, 8, 4, 2}
    checkCodes := []byte{'1', '0', 'X', '9', '8', '7', '6', '5', '4', '3', '2'}

    sum := 0
    for i := 0; i < 17; i++ {
        sum += int(idCard[i]-'0') * weights[i]
    }
    return idCard[17] == checkCodes[sum%11]
}
```

#### 7.3.2 角色申请状态机

```
状态流转图：

                    ┌──────────┐
                    │ 未申请   │
                    └────┬─────┘
                         │ 提交申请
                         ▼
                    ┌──────────┐
            ┌──────│ pending  │──────┐
            │      │  (待审核) │      │
            │      └──────────┘      │
            │                        │
       审核驳回                  审核通过
            │                        │
            ▼                        ▼
      ┌──────────┐            ┌──────────┐
      │ rejected │            │ approved │
      │  (已驳回) │            │  (已通过) │
      └────┬─────┘            └────┬─────┘
           │                       │
      重新提交               管理员撤销
           │                       │
           ▼                       ▼
      ┌──────────┐            ┌──────────┐
      │ pending  │            │ revoked  │
      └──────────┘            │  (已撤销) │
                              └────┬─────┘
                                   │
                              30天后可重新申请
                                   │
                                   ▼
                              ┌──────────┐
                              │ pending  │
                              └──────────┘
```

**状态流转规则**：

| 当前状态 | 目标状态 | 触发条件 | 约束 |
|---------|---------|---------|------|
| 无记录 | pending | 用户提交申请 | 已实名、无 pending 申请、冷却期已过 |
| pending | approved | 管理员审批通过 | 管理员/超级管理员操作 |
| pending | rejected | 管理员审批驳回 | 需填写驳回原因 |
| approved | revoked | 管理员撤销角色 | 管理员/超级管理员操作 |
| rejected | pending | 用户重新提交 | 无冷却期限制 |
| revoked | pending | 用户重新申请 | 撤销后 30 天冷却期 |

**并发控制**：

- 使用数据库唯一索引 `(user_id, role_id, status)` + 应用层检查，防止同一角色同时存在多个 pending 申请
- 管理员审核操作使用乐观锁（WHERE status=0），防止重复审核

```

   │   - page_size: 每页条数，默认 10，最大 100

   └── 调用 Service 层 GetUserList(request)

2. Service 层 (user_admin_service.go)
   ├── 2.1 构建查询条件
   │   ├── WHERE deleted_at IS NULL
   │   ├── AND keyword LIKE ? (若 keyword 非空，匹配 username/email/phone/nickname)
   │   ├── AND status = ? (若 status 非空)
   │   ├── AND created_at BETWEEN ? AND ? (若时间范围非空)
   │   └── AND id IN (SELECT user_id FROM user_roles WHERE role_id=?) (若 role_id 非空)

   │   ├── 全部数据权限 → 无额外过滤

   │   ├── 本部门 → AND dept_id = 用户所在部门
   │   └── 仅本人 → AND id = 当前用户 ID
   │
   ├── 2.3 分页查询
   │   ├── Repository: SELECT COUNT(*) FROM users WHERE ... → total
   │   └── Repository: SELECT * FROM users WHERE ... ORDER BY created_at DESC LIMIT ? OFFSET ?
   │
   ├── 2.4 批量查询角色信息
   │   ├── Repository: SELECT ur.user_id, r.* FROM user_roles ur JOIN roles r ON ur.role_id=r.id WHERE ur.user_id IN (?)
   │   └── 组装到每个用户的 roles 字段
   │
   ├── 2.5 组装响应
   │   └── { list: [UserInfo], total, page, page_size }
   │
   └── 2.6 返回分页数据

3. Controller 层
   └── response.Success(c, pageResult)
```

**查询优化**：

- 使用 `(status, deleted_at)` 联合索引加速筛选
- 关键字搜索使用 `LIKE '%keyword%'`（无法使用索引，数据量大时考虑全文索引或 Elasticsearch）
- 角色筛选使用子查询 `id IN (SELECT user_id FROM user_roles WHERE role_id=?)`，`user_roles` 表有 `(user_id, role_id)` 联合索引
- 分页使用 `LIMIT offset, size`，大偏移量时考虑游标分页

### 7.5 隐私设置

#### 7.5.1 隐私设置模型

**可见性级别定义**：

| 值 | 含义 | 说明 |
|---|------|------|
| 1 | 所有人可见 | 默认值（手机号除外） |
| 2 | 仅班级成员可见 | 加入同一班级的成员可查看 |
| 3 | 仅自己可见 | 只有本人可查看 |

**默认值**：

| 设置项 | 默认值 | 说明 |
|--------|-------|------|
| profile_visible | 1 (所有人) | 个人主页可见性 |
| realname_visible | 1 (所有人) | 真实姓名可见性 |
| email_visible | 1 (所有人) | 邮箱可见性 |
| stats_visible | 1 (所有人) | 学习统计可见性 |
| class_visible | 1 (所有人) | 班级信息可见性 |

**注意**：手机号始终为"仅自己"可见，不提供修改选项。

#### 7.5.2 隐私设置读取与更新

**读取**：`GET /api/v1/user/privacy`

```
1. Controller → Service → Repository
   ├── Repository: SELECT * FROM user_privacy WHERE user_id=?
   ├── 若无记录 → 创建默认记录并返回
   └── 返回 PrivacyResponse DTO
```

**更新**：`PUT /api/v1/user/privacy`

```
1. Controller 层
   ├── 绑定 UpdatePrivacyRequest DTO
   │   - profile_visible: 可选, 1/2/3
   │   - realname_visible: 可选, 1/2/3
   │   - email_visible: 可选, 1/2/3
   │   - stats_visible: 可选, 1/2/3
   │   - class_visible: 可选, 1/2/3
   └── 调用 Service 层 UpdatePrivacy(userID, request)

2. Service 层
   ├── 值校验：各字段值必须为 1/2/3
   ├── Repository: UPDATE user_privacy SET ... WHERE user_id=?
   │   └── 若记录不存在 → INSERT
   └── 清除该用户的隐私缓存（Redis DEL qh:user:privacy:{user_id}）
```

#### 7.5.3 隐私控制在数据查询中的应用

当查询其他用户资料时（如查看班级成员信息），需要根据隐私设置过滤返回字段：

```
Service: GetUserProfileWithPrivacy(viewerID, targetUserID)
    │
    ├── 1. 查询目标用户资料
    │
    ├── 2. 查询目标用户隐私设置
    │   └── 优先从 Redis 缓存读取: qh:user:privacy:{targetUserID}
    │
    ├── 3. 判断查看者与目标用户的关系
    │   ├── viewerID == targetUserID → 本人，返回全部信息
    │   ├── 是否为同一班级成员 → 调用 class service 检查
    │   └── 其他 → 普通用户
    │
    ├── 4. 根据隐私设置过滤字段
    │   ├── profile_visible=3 且非本人 → 返回 403 或仅返回基础信息
    │   ├── realname_visible=2 且非班级成员 → 隐藏真实姓名
    │   ├── email_visible=3 且非本人 → 隐藏邮箱
    │   ├── stats_visible=2 且非班级成员 → 隐藏学习统计
    │   └── class_visible=3 且非本人 → 隐藏班级信息
    │
    └── 5. 返回过滤后的用户资料
```

```go
    // 角色申请审核
    adminGroup.GET("/applications", middleware.RequirePermission("system:application:list"), adminApplicationController.List)
    adminGroup.POST("/applications/:id/approve", middleware.RequirePermission("system:application:review"), adminApplicationController.Approve)
}
```

**数据权限中间件**：

```
DataPermissionMiddleware()
    │
    ├── 1. 获取当前用户的数据权限范围
    │   ├── 查询 data_permissions 表
    │   └── 获取 type 字段（1=全部 2=本部门及下级 3=本部门 4=仅本人）
    │
    ├── 2. 根据数据权限类型设置查询条件
    │   ├── type=1 (全部) → 不添加额外条件
    │   ├── type=2 (本部门及下级) → c.Set("dept_ids", visibleDeptIDs)
    │   ├── type=3 (本部门) → c.Set("dept_id", userDeptID)
    │   └── type=4 (仅本人) → c.Set("owner_only", true)
    │
    └── 3. c.Next() 继续处理
        └── Controller/Service 层从上下文读取数据权限条件，拼接到查询中
```

## 九、安全考虑

### 9.1 身份证号安全

| 措施 | 说明 |
|------|------|
| AES 加密存储 | 身份证号使用 AES-256-CBC 加密后存储，密钥独立管理（配置文件 aes_key 字段） |
| SHA-256 哈希索引 | 使用 id_card_hash 字段做唯一性校验，避免解密比对 |
| API 响应脱敏 | 返回给前端的身份证号格式：`110***********1234`（前 3 后 4） |
| 日志脱敏 | 日志中记录身份证号时自动脱敏 |
| 禁止精确搜索 | 查询接口不支持按身份证号精确搜索，仅管理员审核时可查看 |

### 9.2 隐私保护

| 措施 | 说明 |
|------|------|
| 多级可见性 | 个人主页、真实姓名、邮箱、统计、班级信息均支持三级可见性控制 |
| 查询时过滤 | 查看他人资料时，根据隐私设置动态过滤返回字段 |
| 缓存策略 | 隐私设置变更后立即清除 Redis 缓存，避免旧配置泄露 |
| 手机号保护 | 手机号始终仅自己可见，不提供修改选项 |

### 9.3 角色申请安全

| 措施 | 说明 |
|------|------|
| 实名前置 | 申请创作者/教师角色前必须完成实名认证 |
| 唯一 pending | 同一角色同时只能有一个待审核申请，防止重复提交 |
| 撤销冷却期 | 角色被撤销后 30 天内不可重新申请，防止滥用 |
| 系统角色保护 | 系统角色 (is_system=true) 不可申请，仅管理员手动分配 |
| 审计日志 | 所有角色分配、撤销操作记录审计日志 |

### 9.4 管理端安全

| 措施 | 说明 |
|------|------|
| 权限细分 | 用户管理拆分为 list/query/add/edit/delete/status/role/resetpwd/export 等独立权限 |
| 禁止自我操作 | 管理员不能禁用自己、不能删除自己 |
| 操作日志 | 管理端所有写操作通过 OperationLog 中间件记录 |
| 批量限制 | 批量操作限制单次最多 100 条，防止误操作 |
| 密码重置安全 | 重置后全设备退出，临时密码仅返回一次 |

### 9.5 输入安全

| 措施 | 说明 |
|------|------|
| 参数校验 | 使用 Gin binding tags 进行格式、长度、范围校验 |
| 敏感词过滤 | 昵称、简介、申请理由等用户输入经过 DFA 敏感词过滤器 |
| SQL 注入防护 | GORM 参数化查询，杜绝 SQL 注入 |
| XSS 防护 | 输出转义，Content-Type: application/json |
| 文件上传安全 | 头像上传校验 MIME type、文件大小、图片尺寸 |
