## 一、模块概述

评论社区模块是题小助平台的社交互动核心，涵盖以下功能域：

- **多态评论系统**：支持对题目（target_type=1）、考试（target_type=2）、班级（target_type=3）三种实体发表评论，通过 (target_type, target_id) 多态关联实现一套评论代码服务多种业务对象
- **评论树（嵌套回复）**：通过 parent_id 实现楼中楼回复，最多 2 层嵌套；查询时在 Service 层构建评论树返回前端
- **评论互动**：点赞/取消点赞（唯一索引防重复）、图片上传（最多 9 张）、@提及用户、表情包支持
- **评论管理**：置顶、精选、官方解答标记、批量审核、批量删除、评论导出
- **敏感词过滤**：基于 DFA 算法的实时内容过滤，评论创建/编辑时自动检查，中间件层拦截
- **举报工作流**：用户举报 -> 管理员审核 -> 处理动作（维持/隐藏/警告/禁言），含举报去重和自举报拦截
- **评论黑名单**：支持全局禁言和目标禁言（per-target），可设置过期时间
- **审核规则引擎**：关键词/正则/长度/重复等规则类型，支持自动标记待审、自动隐藏、自动删除等动作
- **通知系统**：WebSocket 实时推送 + 数据库持久化，支持 11 种通知类型，含通知偏好设置、免打扰、群发、定时发送、撤回、模板管理

**评论请求处理流程**：

```
客户端提交评论
    │
    ▼
Router (路由匹配: POST /api/v1/comments)
    │
    ▼
Middleware 链
    ├── CORS 中间件
    ├── RateLimit 中间件 (IP 限流)
    ├── Auth 中间件 (JWT 解析 + 黑名单检查)
    └── SensitiveFilter 中间件 (请求体敏感词扫描)
    │
    ▼
Controller (CommentController.CreateComment)
    ├── 绑定 CreateCommentRequest DTO
    ├── 参数校验 (target_type, target_id, content)
    └── 调用 Service 层
    │
    ▼
Service (comment.CreateComment)
    ├── 1. 黑名单检查 (全局 + 目标级)
    ├── 2. 敏感词检查 (DFA 算法)
    ├── 3. 审核规则匹配 (关键词/正则/长度/重复)
    ├── 4. 构建 Comment model
    ├── 5. Repository 写入数据库
    └── 6. 触发通知 (WebSocket 推送 + DB 持久化)
    │
    ▼
Repository (comment.Create)
    └── GORM INSERT INTO comments
```

## 四、涉及文件

### 后端文件

**路由层 (Router)**：

- `internal/router/router.go` — 路由总入口，实例化 CommentController、CommentAdminController、NotificationController、NotificationAdminController，注册路由组
- `internal/router/comment.go` — 评论路由定义：用户端 `/api/v1/comments/*`、表情包 `/api/v1/sticker/*`、用户搜索 `/api/v1/user/search`；管理端 `/api/v1/admin/comments/*`（含置顶/精选/官方解答/批量操作/举报/黑名单/审核规则）

**控制器层 (Controller)**：

- `internal/controller/comment/comment_controller.go` — 用户端评论控制器：ListComments、CreateComment、EditComment、DeleteComment、LikeComment、UploadImage、ListStickers、ListStickerCategories、SearchUsers
- `internal/controller/comment/comment_admin_controller.go` — 管理端评论控制器：PinComment、UnpinComment、FeatureComment、UnfeatureComment、MarkOfficial、UnmarkOfficial、ListBlacklists、AddBlacklist、RemoveBlacklist、ListAuditRules、CreateAuditRule、UpdateAuditRule、DeleteAuditRule、BatchAudit、BatchDelete、GetCommentStats、ExportComments

- `internal/controller/notification/notification_controller.go` — 用户端通知控制器：ListNotifications、GetUnreadCount、MarkAsRead、MarkAllAsRead、DeleteNotification、RecallNotification、BatchSend、BatchMarkAsRead、BatchDeleteNotifications、CreateScheduled、ListScheduled、DeleteScheduled、GetNotificationSettings、UpdateNotificationSettings、GetNotificationStats
- `internal/controller/notification/notification_admin_controller.go` — 管理端通知控制器：ListTemplates、CreateTemplate、UpdateTemplate、DeleteTemplate、ListChannels、UpdateChannel

**业务逻辑层 (Service)**：

- `internal/service/comment/comment_service.go` — 评论核心业务：列表查询（含树构建）、创建（黑名单+敏感词检查）、编辑、删除、点赞（事务原子性）、举报（去重检查）、置顶/精选/官方解答、表情包、用户搜索、黑名单 CRUD、审核规则 CRUD、举报处理（含自动隐藏）、批量操作、统计、导出
- `internal/service/notification/notification_service.go` — 通知核心业务：列表查询、未读计数、已读标记（单条/批量/全部）、删除、撤回、群发（按角色/班级/用户组/全部）、定时通知、通知设置（偏好+免打扰）、通知统计、模板 CRUD、渠道管理
- `internal/service/sensitive/sensitive_service.go` — 敏感词管理：列表、创建、删除、导入、测试

**数据访问层 (Repository)**：

- `internal/repository/comment/comment_repo.go` — 评论数据访问：Comment CRUD、CommentLike 点赞操作（含原子计数增减）、CommentReport 举报操作、CommentBlacklist 黑名单操作、Sticker 表情包查询、CommentAuditRule 审核规则操作、用户搜索、统计查询（按状态/日期/目标）、导出查询

- `internal/model/notification_ext.go` — NotificationTemplate（通知模板，含编码、标题/内容模板、类型、渠道、启用状态）、NotificationSetting（通知设置，按用户+类型+渠道唯一，含免打扰时段）、ScheduledNotification（定时通知，含计划时间、状态、批次ID）

## 五、数据库表

评论社区模块涉及以下数据表（共 9 张）：

| 表名 | 说明 | 关键字段 | 关联关系 |
|------|------|----------|----------|
| `comments` | 评论主表 | id, target_type (1题目/2考试/3班级), target_id, user_id (FK→users), content, parent_id (自引用), images, mentions, reply_to_user_id, like_count, is_pinned, is_featured, is_official, status (0隐藏/1正常/2举报) | 多态关联：(target_type, target_id) 关联 questions/exams/classes；user_id FK→users；parent_id 自引用 |
| `comment_likes` | 评论点赞表 | id, comment_id (FK→comments), user_id (FK→users), created_at | 唯一索引 (comment_id, user_id) 防重复点赞 |
| `comment_reports` | 评论举报表 | id, comment_id (FK→comments), user_id (FK→users), reason, reason_type (spam/abuse/infringement/other), status (0待处理/1已处理/2已驳回), handler_id, handle_remark, handled_at | N:1 关联 comments 和 users |
| `notifications` | 通知表 | id, user_id (FK→users), sender_id, type (1系统/2考试/3作业/4班级/5评论), title, content, target_type, target_id, is_read, is_recalled, recalled_at, channel (app/email/sms/wechat), batch_id | N:1 关联 users；batch_id 用于群发通知分组 |
| `comment_blacklist` | 评论黑名单表 | id, user_id (FK→users), target_type (0全局/1题目/2考试/3班级), target_id, reason, operator_id, expires_at | 唯一索引 (user_id, target_type, target_id)；全局禁言时 target_type=0, target_id=0 |
| `notification_settings` | 通知设置表 | id, user_id (FK→users), type, channel, enabled, do_not_disturb, disturb_start (HH:mm), disturb_end (HH:mm) | 唯一索引 (user_id, type, channel)；Upsert 语义 |
| `stickers` | 表情包表 | id, name, category, image_url, sort_order, status (0禁用/1启用) | 独立表，按 category 分组 |
| `notification_templates` | 通知模板表 | id, code (unique), name, title_tpl, content_tpl, type, channel, is_active, remark | 模板变量使用 `{{.Var}}` 语法 |
| `comment_audit_rules` | 评论审核规则表 | id, name, rule_type (keyword/regex/length/repeat), pattern, action (1标记待审/2自动隐藏/3自动删除/4拒绝发布), priority, description, status | 按 priority DESC 排序匹配 |

### 7.2 评论树

**设计目标**：支持楼中楼回复，最多 2 层嵌套，查询时返回树形结构。

**数据模型**：

```go
type Comment struct {
    ParentID      *uint  `gorm:"index" json:"parent_id"`       // 父评论 ID，nil 表示顶级评论
    ReplyToUserID *uint  `gorm:"index" json:"reply_to_user_id"` // 回复的目标用户 ID
    // ...
}
```

**树构建算法**（Service 层 `buildCommentTree`）：

```
输入: comments []CommentInfo（平铺列表，已按 is_pinned DESC, created_at DESC 排序）

步骤:
1. 按 ParentID 分组到 childMap: map[uint][]CommentInfo
   - 遍历所有评论
   - 如果 ParentID != nil，将其加入 childMap[*ParentID]

2. 构建树
   - 遍历所有评论
   - 如果 ParentID == nil（顶级评论）：
     - 从 childMap 中取出子评论赋值给 Children 字段
     - 加入结果列表

输出: tree []CommentInfo（仅包含顶级评论，子评论嵌套在 Children 中）
```

**查询优化**：

```
当前实现（单次查询 + 内存构建）：
  SELECT * FROM comments
  WHERE target_type = ? AND target_id = ? AND status = 1
  ORDER BY is_pinned DESC, created_at DESC
  LIMIT ?, ?

  优点：一次 SQL 查询，简单高效
  适用：单个目标下的评论数量通常在百级以内

优化方向（评论量大时）：
  - 只查询顶级评论（parent_id IS NULL），分页
  - 按需懒加载子评论（点击"查看回复"时异步加载）
  - Redis 缓存热门目标的评论树
```

**回复关系**：

```
顶级评论 (parent_id = nil)
  └── 子评论 (parent_id = 顶级评论ID, reply_to_user_id = 被回复用户ID)
        └── 不再嵌套（最多 2 层）

前端展示：
  - 顶级评论显示完整内容
  - 子评论显示 "回复 @用户名：内容"
  - 通过 reply_to_user_id 关联被回复用户信息
```

### 7.3 敏感词过滤

**实现位置**：`pkg/sensitive/filter.go` + `internal/middleware/sensitive.go`

**当前实现（map 匹配）**：

```go
// Filter 结构体
type Filter struct {
    words map[string]bool      // 敏感词集合
    mu    sync.RWMutex         // 读写锁，线程安全
}

// HasSensitive 检查是否包含敏感词
func (f *Filter) HasSensitive(content string) bool {
    f.mu.RLock()
    defer f.mu.RUnlock()
    for word := range f.words {
        if strings.Contains(content, word) {
            return true
        }
    }
    return false
}
```

**调用时机**（双重过滤）：

```
第一层：中间件拦截 (middleware/sensitive.go)
    ├── 拦截所有 POST/PUT 请求
    ├── 读取请求体全文
    ├── filter.HasSensitive(body)
    └── 命中 → 返回 400 "内容包含敏感词，请修改后重试"

第二层：Service 层精确检查 (comment_service.go)
    ├── CreateComment 和 EditComment 时
    ├── 仅检查 content 字段
    ├── filter.HasSensitive(req.Content)
    └── 命中 → 返回错误
```

**优化方向（DFA 算法）**：

```
当前 map 匹配的时间复杂度为 O(n*m)，n=内容长度，m=敏感词数量。
对于大量敏感词场景，建议升级为 DFA（确定有限自动机）算法：

1. 构建阶段：
   - 将所有敏感词构建成 Trie 树
   - 每个节点的失败指针指向最长后缀匹配节点
   - 时间复杂度：O(Σ|word|)

2. 匹配阶段：
   - 遍历内容字符串，沿 DFA 状态转移
   - 时间复杂度：O(|content|)，与敏感词数量无关

3. 优化效果：
   - 1000 个敏感词时，匹配速度提升 100x+
   - 支持敏感词热更新（重建 DFA 树）
```

**敏感词来源**：

```
1. 数据库 sensitive_words 表（通过 service/sensitive 管理）
2. 配置文件内置敏感词列表
3. 运行时通过管理端 API 动态添加/删除

当前实现：filter 在每次调用时创建新实例并加载默认词库。
优化方向：全局单例 + 定时从数据库同步 + 热更新通知。
```

### 7.6 WebSocket 实现

**架构设计**：

```
┌─────────────────────────────────────────────────────────┐
│                     WebSocket Hub                        │
│                                                          │
│  clients: map[uint]map[*Client]bool                     │
│           ┌─────────────────────────────────┐           │
│           │  UserID=1 → [ClientA, ClientB]  │  (多设备) │
│           │  UserID=2 → [ClientC]           │           │
│           │  UserID=3 → [ClientD, ClientE]  │  (多设备) │
│           └─────────────────────────────────┘           │
│                                                          │
│  Register   chan *Client   ← 新连接注册                  │
│  Unregister chan *Client   ← 连接断开注销                │
│  Broadcast  chan *Message  ← 消息广播                    │
│                                                          │
│  Run() → select {                                        │
│      case client := <-Register:   register(client)       │
│      case client := <-Unregister: unregister(client)     │
│      case message := <-Broadcast: broadcast(message)     │
│  }                                                       │
└─────────────────────────────────────────────────────────┘
```

**Hub 核心逻辑**（`internal/ws/hub.go`）：

```go
// Hub 连接管理中心
type Hub struct {
    clients    map[uint]map[*Client]bool  // userID → 客户端集合（支持多设备）
    Register   chan *Client
    Unregister chan *Client
    Broadcast  chan *Message
    mu         sync.RWMutex
}

// register 注册客户端
func (h *Hub) register(client *Client) {
    h.mu.Lock()
    defer h.mu.Unlock()
    if _, ok := h.clients[client.UserID]; !ok {
        h.clients[client.UserID] = make(map[*Client]bool)
    }
    h.clients[client.UserID][client] = true
}

// broadcast 广播消息
func (h *Hub) broadcast(message *Message) {
    h.mu.RLock()
    defer h.mu.RUnlock()

    if message.UserID > 0 {
        // 定向发送给指定用户的所有设备
        if clients, ok := h.clients[message.UserID]; ok {
            for client := range clients {
                select {
                case client.Send <- message:
                default:
                    close(client.Send)
                    delete(clients, client)
                }
            }
        }
    } else {
        // 广播给所有在线用户
        for _, clients := range h.clients {
            for client := range clients {
                select {
                case client.Send <- message:
                default:
                    close(client.Send)
                    delete(clients, client)
                }
            }
        }
    }
}

// SendToUser 发送消息给指定用户
func (h *Hub) SendToUser(userID uint, message *Message) {
    message.UserID = userID
    h.Broadcast <- message
}

// IsOnline 检查用户是否在线
func (h *Hub) IsOnline(userID uint) bool {
    h.mu.RLock()
    defer h.mu.RUnlock()
    clients, ok := h.clients[userID]
    return ok && len(clients) > 0
}
```

**Client 核心逻辑**（`internal/ws/client.go`）：

```go
// 连接参数
const (
    writeWait      = 10 * time.Second   // 写超时
    pongWait       = 60 * time.Second   // 等待 Pong 超时
    pingPeriod     = 54 * time.Second   // 发送 Ping 间隔 (pongWait * 9/10)
    maxMessageSize = 4096               // 最大消息大小
)

// Client WebSocket 客户端
type Client struct {
    Hub    *Hub
    UserID uint
    Conn   *websocket.Conn
    Send   chan *Message  // 缓冲通道，容量 256
}

// ReadPump 读取消息（独立 goroutine）
func (c *Client) ReadPump() {
    defer func() {
        c.Hub.Unregister <- c
        c.Conn.Close()
    }()

    c.Conn.SetReadLimit(maxMessageSize)
    c.Conn.SetReadDeadline(time.Now().Add(pongWait))
    c.Conn.SetPongHandler(func(string) error {
        c.Conn.SetReadDeadline(time.Now().Add(pongWait))
        return nil
    })

    for {
        _, message, err := c.Conn.ReadMessage()
        if err != nil {
            break
        }
        msg := &Message{UserID: c.UserID, Data: message}
        c.Hub.Broadcast <- msg
    }
}

// WritePump 写入消息（独立 goroutine）
func (c *Client) WritePump() {
    ticker := time.NewTicker(pingPeriod)
    defer func() {
        ticker.Stop()
        c.Conn.Close()
    }()

    for {
        select {
        case message, ok := <-c.Send:
            c.Conn.SetWriteDeadline(time.Now().Add(writeWait))
            if !ok {
                c.Conn.WriteMessage(websocket.CloseMessage, []byte{})
                return
            }
            if err := c.Conn.WriteJSON(message); err != nil {
                return
            }
        case <-ticker.C:
            c.Conn.SetWriteDeadline(time.Now().Add(writeWait))
            if err := c.Conn.WriteMessage(websocket.PingMessage, nil); err != nil {
                return
            }
        }
    }
}
```

2. 后端验证
   ├── 解析 JWT Token
   ├── 验证用户身份
   └── 创建 Client 实例

3. 注册到 Hub
   └── Hub.Register <- client

4. 启动读写 goroutine
   ├── go client.ReadPump()   // 读取客户端消息
   └── go client.WritePump()  // 推送服务端消息

5. 心跳保活
   ├── 服务端每 54s 发送 Ping
   ├── 客户端回复 Pong
   └── 60s 未收到 Pong → 断开连接

6. 连接断开
   ├── ReadPump 退出 → Hub.Unregister <- client
   └── Hub 清理客户端映射，关闭 Send 通道
```

### 8.1 敏感词过滤中间件

```
SensitiveFilterMiddleware()
    │
    ├── 仅拦截 POST/PUT 请求
    │
    ├── 读取请求体 (io.ReadAll)
    │

## 九、Redis 缓存设计

### 9.1 评论模块缓存 Key

| Key Pattern | 数据类型 | TTL | 说明 |
|-------------|---------|-----|------|
| `qh:comment:count:{target_type}:{target_id}` | String (INCR) | 3600s | 目标评论计数缓存 |
| `qh:comment:like:{comment_id}:{user_id}` | String | 86400s | 用户点赞状态缓存（减少 DB 查询） |
| `qh:comment:tree:{target_type}:{target_id}` | String (JSON) | 300s | 评论树缓存（热门目标） |
| `qh:sensitive:words` | String (JSON) | 3600s | 敏感词库缓存（定期从 DB 同步） |

### 9.2 通知模块缓存 Key

| Key Pattern | 数据类型 | TTL | 说明 |
|-------------|---------|-----|------|
| `qh:notification:unread:{user_id}` | String (INCR) | 不过期 | 用户未读通知计数（增量更新） |
| `qh:notification:settings:{user_id}` | Hash | 3600s | 用户通知偏好设置缓存 |
| `qh:notification:daily:{user_id}` | String (INCR) | 86400s | 用户每日通知发送计数（防刷） |

### 9.3 缓存更新策略

```
评论计数缓存：
  ├── 创建评论 → INCR qh:comment:count:{type}:{id}
  ├── 删除评论 → DECR qh:comment:count:{type}:{id}
  └── 缓存未命中 → 从 DB 查询并回填

未读通知计数：
  ├── 创建通知 → INCR qh:notification:unread:{uid}
  ├── 标记已读 → DECR qh:notification:unread:{uid}
  ├── 全部已读 → DEL qh:notification:unread:{uid}
  └── 缓存未命中 → 从 DB 查询 COUNT 并回填

通知偏好缓存：
  ├── 更新设置 → DEL qh:notification:settings:{uid}（失效）
  ├── 查询时先读缓存 → 未命中则查 DB 并回填
  └── TTL 1 小时，自动过期兜底
```

## 十、安全考虑

### 10.1 评论内容安全

| 措施 | 说明 |
|------|------|
| 敏感词过滤 | 中间件层 + Service 层双重过滤，DFA 算法实时检查 |
| 内容长度限制 | 评论内容最长 500 字符（DTO binding: `max=500`） |
| 图片数量限制 | 最多 9 张图片（Service 层 validateAndCleanImages） |
| HTML 净化 | 评论内容禁止 script、事件属性等危险 HTML |
| 举报机制 | 用户可举报不当评论，举报后进入待审核状态 |
| 自动审核规则 | 支持关键词/正则/长度/重复等规则，触发后自动标记/隐藏/删除 |

### 10.2 评论权限安全

| 措施 | 说明 |
|------|------|
| 编辑/删除限制 | 只能操作自己的评论（Service 层 UserID 校验） |
| 黑名单机制 | 被禁言用户无法发表评论（全局或目标级） |
| 黑名单过期 | 支持设置过期时间，过期后自动恢复评论权限 |
| 管理员权限 | 管理端接口需 JWT + AdminOnly 中间件双重校验 |
| 举报去重 | 同一用户对同一评论只能举报一次（数据库查询去重） |
| 自举报拦截 | 不能举报自己的评论 |

### 10.3 通知安全

| 措施 | 说明 |
|------|------|
| 权限校验 | 标记已读/删除操作校验 UserID，防止越权操作 |
| 撤回限制 | 只有未读通知可撤回，已读通知不可撤回 |
| 群发限制 | 群发通知需管理员权限，普通用户只能发送目标通知 |
| 通知频率限制 | Redis 计数器限制每日通知发送次数，防止通知轰炸 |
| 免打扰 | 支持设置免打扰时段，时段内不推送通知 |

### 10.4 WebSocket 安全

| 措施 | 说明 |
|------|------|
| Token 认证 | WebSocket 连接需携带 JWT Token，服务端验证 |
| 消息大小限制 | maxMessageSize = 4096 字节 |
| 心跳检测 | 60s 未收到 Pong 自动断开，防止僵尸连接 |
| 缓冲通道 | Send 通道容量 256，满时关闭连接，防止内存溢出 |
| 定向发送 | 消息按 UserID 定向推送，不会泄露给其他用户 |
