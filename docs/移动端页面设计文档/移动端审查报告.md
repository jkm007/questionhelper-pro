## 二、页面设计合理性审查

### 2.1 设计质量总评

所有 14 个目录的页面设计文档均存在**系统性缺陷**，主要表现为以下几点：

1. **API 接口定义缺失**：全部页面设计文档均未列出具体的 API 端点路径、请求参数和响应格式，仅使用函数名引用（如 `getBanners`、`getUserStatistics`），与后端实际路由（`/api/v1/...`）无法直接对应。
2. **异常状态覆盖不足**：加载中（skeleton/loading）、空状态（empty state）、网络失败（error state）三种基础状态在绝大多数页面中缺失。
3. **交互细节不够具体**：防重复提交、表单校验规则、分页参数、键盘适配等细节普遍缺失或过于简略。

API 模块文件齐全（13 个），但存在以下问题：

| 问题 | 文件 | 说明 |
|------|------|------|
| **硬编码 localhost URL** | `api/file.ts`、`api/user.ts` | `uploadFile()`、`uploadImage()`、`uploadAvatar()` 使用 `http://localhost:8080` 而非常量 `BASE_URL` |

### 7.3 关键风险提示

1. **班级模块缺口最大**：18 个设计页面仅实现 8 个（44%），是所有模块中完成度最低的。建议优先补齐作业管理全流程（布置/提交/批改/结果），因为这是师生互动的核心链路。
2. **API 层存在运行时 BUG**：硬编码 URL 和响应码类型不一致是阻塞性问题，必须在任何功能开发前修复。
3. **WebSocket 是多个功能的前置依赖**：通知系统、考试提醒、作业通知、班级公告推送均依赖 WebSocket，建议在阶段 2 优先完成。
4. **页面设计文档需同步更新**：注册页设计与后端 DTO 严重不匹配、通知类型不一致等问题需要先修正设计文档，再进行开发，避免返工。
5. **API 重复函数需尽早清理**：当前 13 个 API 文件中存在约 16 对重复函数，随着功能开发继续会产生更多歧义，建议在阶段 2 统一处理。
