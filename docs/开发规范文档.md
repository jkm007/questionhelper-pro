## 一、Git 分支策略

### 分支模型

采用 **Git Flow** 简化版：

```
main (生产) ─────────────────────────────────────────►
  │
  ├── develop (开发) ────────────────────────────────►
  │     │
  │     ├── feature/xxx (功能分支)
  │     ├── feature/yyy
  │     └── ...
  │
  ├── release/x.x.x (发布分支，可选)
  │
  └── hotfix/xxx (紧急修复)
```

### 分支说明

| 分支 | 说明 | 合并目标 |
|------|------|----------|
| `main` | 生产环境代码，始终可部署 | - |
| `develop` | 开发主分支，集成所有功能 | main |
| `feature/*` | 功能分支，从 develop 创建 | develop |
| `hotfix/*` | 紧急修复，从 main 创建 | main + develop |
| `release/*` | 发布准备分支 (可选) | main + develop |

### 分支命名规范

```
feature/题库管理-批量导入功能
feature/exam-anti-cheat
fix/login-captcha-error
hotfix/sql-injection-fix
refactor/user-service-cleanup
docs/api-documentation
```

前缀类型：
- `feature/` - 新功能
- `fix/` - Bug 修复
- `hotfix/` - 紧急修复
- `refactor/` - 重构
- `docs/` - 文档
- `test/` - 测试
- `chore/` - 构建/工具

### 分支操作流程

```bash
# 创建功能分支
git checkout develop
git pull origin develop
git checkout -b feature/新功能名

# 开发完成后合并
git checkout develop
git merge --no-ff feature/新功能名
git push origin develop
git branch -d feature/新功能名

# 发布到生产
git checkout main
git merge --no-ff develop
git tag -a v2.0.0 -m "Release v2.0.0"
git push origin main --tags
```

## 二、Commit 规范

### Commit 消息格式

采用 **Conventional Commits** 规范：

```
<type>(<scope>): <subject>

<body>

<footer>
```

### Type 类型

| Type | 说明 | 示例 |
|------|------|------|
| feat | 新功能 | feat(题库): 添加批量导入功能 |
| fix | Bug 修复 | fix(login): 修复验证码过期问题 |
| docs | 文档更新 | docs(api): 更新接口文档 |
| style | 代码格式 (不影响功能) | style: 统一缩进格式 |
| refactor | 重构 (非新功能非修复) | refactor(user): 重构用户服务层 |
| perf | 性能优化 | perf(query): 优化题目列表查询 |
| test | 测试相关 | test(exam): 添加考试模块单元测试 |
| build | 构建流程、外部依赖变更 | build: 升级 Element Plus 到 2.13 |
| ci | CI/CD 相关 | ci: 添加 GitHub Actions 配置 |
| chore | 其他杂项变更 | chore: 更新 Docker 配置 |
| revert | 回滚 | revert: 回滚 feat(xxx) |
| wip | 开发阶段临时提交 | wip: 题目编辑页面半成品 |

### Scope 范围 (可选)

常用 scope：
- `auth` - 认证模块
- `user` - 用户模块
- `question` - 题库模块
- `exam` - 考试模块
- `class` - 班级模块
- `practice` - 练习模块
- `comment` - 评论模块
- `notification` - 通知模块
- `admin` - 管理后台
- `app` - 移动端
- `server` - 后端
- `db` - 数据库
- `api` - API 接口

### 示例

```bash
# 好的 commit 消息
feat(question): 支持 Excel 批量导入题目
fix(exam): 修复考试时间到自动提交失败的问题
refactor(user): 将用户服务拆分为独立模块
docs: 添加数据库 ER 图文档

# 不好的 commit 消息
update code        # 太模糊
fix bug           # 没有说明修了什么
添加功能           # 应使用英文 type
```

### 提交工具

管理后台项目集成了 Commitizen，使用交互式提交：

```bash
cd questionhelper-admin
pnpm commit
```

### 3.1 项目结构

```

├── cmd/server/main.go          # 入口
├── config/config.yaml          # 配置
├── internal/
│   ├── router/                 # 路由定义

│   ├── service/                # 业务逻辑
│   ├── repository/             # 数据访问
│   ├── dto/                    # 请求/响应结构体
│   ├── model/                  # 数据库模型
│   ├── middleware/              # 中间件
│   ├── ws/                     # WebSocket
│   ├── job/                    # 后台任务
│   └── task/                   # 定时任务
├── pkg/                        # 共享包
├── Makefile
└── go.mod
```

### 3.2 分层职责

| 层 | 职责 | 可依赖 |
|----|------|--------|
| Router | 路由注册、中间件挂载 | Controller |
| Controller | 参数绑定、调用 Service、返回响应 | Service, DTO |
| Service | 业务逻辑、事务管理 | Repository |
| Repository | 数据库查询、GORM 操作 | Model |
| DTO | 请求/响应数据结构、参数校验 | - |
| Model | 数据库表结构定义 | - |

**禁止跨层调用**：Controller 不得直接调用 Repository。

### 3.3 命名规范

```go
    userID := ctx.GetUint("userID")
    pageSize := 10

// 常量：大写下划线

// 数据库表名：复数小写下划线 (GORM 默认)
// users, questions, exam_records
```

### 3.4 错误处理

```go
// 使用统一的错误包
import "questionhelper-server/pkg/errors"

// 定义业务错误
var (
    ErrUserNotFound    = errors.New(404, "用户不存在")
    ErrQuestionExists  = errors.New(409, "题目已存在")
    ErrUnauthorized    = errors.New(401, "未授权")
)

// Service 层返回错误
func (s *UserService) GetUser(id uint) (*User, error) {
    user, err := s.repo.FindByID(id)
    if err != nil {
        return nil, errors.Wrap(err, "查询用户失败")
    }
    if user == nil {
        return nil, ErrUserNotFound
    }
    return user, nil
}

// Controller 层处理错误
func (c *UserController) GetUser(ctx *gin.Context) {
    user, err := c.service.GetUser(id)
    if err != nil {
        response.Error(ctx, err)
        return
    }
    response.Success(ctx, user)
}
```

```vue
    </el-table>
  </div>

<script setup lang="ts">
// 1. 导入
import { ref, onMounted } from 'vue'
import { ElMessage } from 'element-plus'
import { getQuestionList } from '@/api/question'
import type { Question } from '@/types/question'

// 2. 类型定义
interface Props {
  categoryId?: number
}

// 3. Props & Emits
const props = withDefaults(defineProps<Props>(), {
  categoryId: undefined,
})
const emit = defineEmits<{
  (e: 'select', question: Question): void
}>()

// 4. 状态
const loading = ref(false)
const tableData = ref<Question[]>([])

// 5. 方法
const fetchData = async () => {
  loading.value = true
  try {
    const { data } = await getQuestionList({ category_id: props.categoryId })
    tableData.value = data.list
  } catch (error) {
    ElMessage.error('获取题目列表失败')
  } finally {
    loading.value = false
  }
}

// 6. 生命周期
onMounted(() => {
  fetchData()
})
</script>

<style scoped lang="scss">
.question-list {
  padding: 20px;
}
</style>
```

## 七、数据库规范

### 7.1 表命名

- 表名使用复数小写下划线：`users`, `exam_records`, `class_members`
- 关联表使用两个实体名拼接：`user_roles`, `paper_questions`
- 避免使用 MySQL 保留字作为表名或字段名

### 7.2 字段命名

- 字段名使用小写下划线：`user_id`, `created_at`, `is_deleted`
- 布尔字段使用 `is_` 前缀：`is_correct`, `is_read`, `is_real`
- 时间字段使用 `_at` 后缀：`created_at`, `updated_at`, `deleted_at`
- 外键字段使用 `_id` 后缀：`user_id`, `question_id`

### 7.3 必备字段

每张表必须包含：

```sql
id          BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT  -- 主键
created_at  DATETIME DEFAULT CURRENT_TIMESTAMP           -- 创建时间
updated_at  DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP  -- 更新时间
deleted_at  DATETIME NULL DEFAULT NULL                   -- 软删除 (可选)
```

### 7.4 索引规范

- 主键：自动聚簇索引
- 外键：必须建立索引
- 高频查询字段：建立索引
- 组合查询：建立联合索引 (注意最左前缀原则)
- 唯一约束：使用 UNIQUE INDEX

### 7.5 GORM 模型示例

```go
type Question struct {
    ID         uint           `gorm:"primarykey" json:"id"`
    Title      string         `gorm:"type:text;not null" json:"title"`
    Content    string         `gorm:"type:text" json:"content"`
    Type       int8           `gorm:"not null" json:"type"`
    Status     int8           `gorm:"default:0" json:"status"`
    CreatorID  uint           `gorm:"index" json:"creator_id"`
    CreatedAt  time.Time      `json:"created_at"`
    UpdatedAt  time.Time      `json:"updated_at"`
    DeletedAt  gorm.DeletedAt `gorm:"index" json:"-"`
}
```

## 八、Code Review 规范

### 8.1 Review 检查项

#### 通用检查

- [ ] 代码是否符合命名规范
- [ ] 是否有未处理的错误
- [ ] 是否有硬编码的配置值
- [ ] 是否有安全隐患 (SQL 注入、XSS 等)
- [ ] 是否有性能问题 (N+1 查询、大循环等)

#### 后端检查

- [ ] 分层是否正确 (Controller → Service → Repository)
- [ ] 是否有数据库事务遗漏
- [ ] 日志是否充分且格式正确
- [ ] 参数校验是否完整
- [ ] 并发安全是否考虑

#### 前端检查

- [ ] 组件是否可复用
- [ ] 类型定义是否完整
- [ ] 是否有内存泄漏风险 (事件监听清理)
- [ ] 加载状态和错误处理是否完善
- [ ] 是否适配移动端 (响应式)

### 8.2 Review 流程

1. 开发者提交 PR，填写 PR 描述
2. 自动化检查通过 (lint, type-check, test)
3. 至少一位团队成员 Review
4. Review 通过后合并到 develop
5. 合并时使用 `--no-ff` 保留分支历史
